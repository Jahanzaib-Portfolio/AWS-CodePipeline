# AWS CodePipeline - Building Phase (Continuous Integration)

This repository contains the documentation and implementation details for the **Building Phase (Continuous Integration)** of our AWS automated pipeline. In this phase, we set up **AWS CodeBuild** to automatically pull source files from GitHub, provision the build container environments, handle dependency updates, and test code compilation.

---

## 1. CI/CD Architecture Overview
The core automation flow integrates a central version control repository with pipeline actions that trigger a sequence of checkouts, unit testing (UT), security scanning, container image building, and pushing artifacts to a remote registry.

![CI/CD Pipeline Sequence Flow](images/image1)

---

## 2. Setting Up the CodeBuild Workspace
1. Open the AWS Management Console and navigate to the **AWS CodeBuild** service dashboard.
2. From the side navigation pane, click on **Build projects**.
3. Click the orange **Create project** button on the main control panel.

![AWS CodeBuild Dashboard Panel](images/image2)

---

## 3. Configuring Project Metadata
1. **Project name:** Enter a descriptive workspace name (e.g., `Python-Project`).
2. **Project type:** Choose **Default project** to execute standard customizable container build instructions.

![Project Configuration Panel](images/image3)

---

## 4. Setting the Source Code Provider
1. Set the **Source provider** attribute dropdown to **GitHub**.
2. Under the Repository scope, toggle the selection to **Public repository** (or authorize your direct GitHub account profile).
3. Input the destination tracking repository URL: `https://github.com/Jahanzaib-Portfolio/AWS-CodePipeline.git`.

![Source Configuration Settings](images/image4)

---

## 5. Environment Provisioning & Service Role Management
1. **Provisioning model:** Select **On-demand** to allocate compute resources dynamically during active execution blocks.
2. **Environment image:** Choose **Managed image** to leverage official, predefined stable OS runtimes.
3. **Compute:** Set the execution platform configuration to **EC2** optimization layers.
4. **Running mode:** Choose **Instance** to run your compilation parameters directly on the allocated system host.

![Environment Settings Configurations](images/image5)

5. **Operating system:** Choose **Amazon Linux** from the dropdown selection map.
6. **Service role:** Select **New service role** and confirm the auto-filled workspace role name (`Python-Project-service-role-for-CodePipeline`). This role gives the build pipeline the IAM permissions it needs to manage logging actions.

![OS and Service Role Specifications](images/image6)

---

## 6. Defining Buildspec Compilation Instructions
1. Scroll down to expand the **Buildspec** configurations drawer module.
2. Under Build specifications, select the **Insert build commands** radio option to provide inline YAML build recipes.
3. Click **Switch to editor** and insert your instructions mapping parameter stores, environment runtime criteria, pre-build installations (`pip install -r requirements.txt`), and automated script validation hooks.

![Buildspec Embedded Code Editor Console](images/image7)

---

## 7. Execution and Tracking Workspace Logs
1. Click **Create build project** at the bottom of the configuration panel.
2. Once the project dashboard builds completely, click **Start build** to run the initial testing loop.

![Initiating CodeBuild Action Sequences](images/image8)

3. Monitor live standard execution outputs and state changes under the **Build log** tab.

![Reviewing Early Phase Integration Streams](images/image9)
![System Environment Variables Mapping Interface](images/image10)
![Tracking Core Compilation Phase Transitions](images/image11)

---

## 8. Verifying Build Success Status
1. The pipeline will transition through status checks including `SUBMITTED`, `PROVISIONING`, `DOWNLOAD_SOURCE`, `INSTALL`, `PRE_BUILD`, `BUILD`, and `POST_BUILD`.
2. Confirm that all build phase steps display a green **Succeeded** checkmark.

# AWS CodePipeline - Deployment Phase (Continuous Deployment)

In this phase, we configure **AWS CodeDeploy** to automatically deploy application artifacts generated by AWS CodeBuild to Amazon EC2 instances, enabling a fully automated release workflow.

---

## 1. Continuous Deployment Architecture Overview

The deployment workflow integrates source control, build automation, and deployment services to deliver application updates directly to EC2 instances.

![Continuous Deployment Architecture](images/image001)

---

## 2. Preparing the Target EC2 Instance

1. Open the AWS Management Console and navigate to the **Amazon EC2** dashboard.
2. Launch a new EC2 instance using **Amazon Linux 2**.
3. Configure the security group to allow:
   - SSH (Port 22)
   - HTTP (Port 80)
   - HTTPS (Port 443)
4. Verify that the instance is running and accessible through SSH.

![EC2 Instance Setup](images/image002)

---

## 3. Installing the AWS CodeDeploy Agent

1. Connect to the EC2 instance using SSH.
2. Update the operating system packages.

```bash
sudo yum update -y
```

3. Download the CodeDeploy installation package from the AWS regional S3 bucket.

```bash
wget https://bucket-name.s3.region-identifier.amazonaws.com/latest/install
```

4. Install the CodeDeploy agent on the EC2 instance.
5. Verify that the agent service is running successfully.

![CodeDeploy Agent Installation](images/image003)

### Important Note

The S3 bucket URL differs depending on the AWS Region. Refer to the AWS documentation to identify the correct regional endpoint.

---

## 4. Configuring EC2 Permissions

1. Create or select an IAM role for the EC2 instance.
2. Attach permissions that allow communication with:
   - AWS CodeDeploy
   - Amazon S3
3. Attach the IAM role to the EC2 instance.
4. Confirm that the permissions are correctly applied.

![EC2 IAM Role Configuration](images/image004)

---

## 5. Restarting and Verifying the CodeDeploy Agent

After completing updates and configuration changes:

1. Restart the CodeDeploy agent service.

```bash
sudo service codedeploy-agent restart
```

2. Verify the service status.

```bash
sudo service codedeploy-agent status
```

3. Ensure the agent status displays **Running**.

![Agent Service Verification](images/image005)

---

## 6. Creating the CodeDeploy Service Role

1. Open the AWS IAM Console.
2. Create a new IAM role for AWS CodeDeploy.
3. Select **AWS Service** as the trusted entity.
4. Choose **CodeDeploy** as the service use case.
5. Attach the required AWS-managed policies.
6. Complete role creation and save the role name.

![CodeDeploy Service Role Creation](images/image006)

---

## 7. Creating the CodeDeploy Application

1. Open the AWS CodeDeploy Console.
2. Navigate to **Applications**.
3. Click **Create application**.
4. Enter a descriptive application name.
5. Select **EC2/On-Premises** as the compute platform.
6. Create the application.

![CodeDeploy Application Configuration](images/image007)

---

## 8. Creating the Deployment Group

1. Open the newly created CodeDeploy application.
2. Click **Create Deployment Group**.
3. Enter a deployment group name.
4. Select the CodeDeploy service role created earlier.
5. Configure deployment targets using EC2 tags.
6. Save the deployment group configuration.

![Deployment Group Configuration](images/image008)

---

## 9. Configuring EC2 Tags

1. Open the target EC2 instance.
2. Navigate to the **Tags** section.
3. Create a tag that identifies the deployment target.
4. Use the same tag key and value inside the Deployment Group configuration.

![EC2 Tag Configuration](images/image009)

### Important Note

Deployment will fail if:

- The EC2 tag does not match the Deployment Group tag.
- CodeDeploy cannot identify the target EC2 instance.

---

## 10. Creating the AppSpec File

1. Create an `appspec.yml` file in the root directory of the GitHub repository.
2. Define deployment instructions and file mappings.
3. Commit and push the file to the repository.

Example:

```yaml
version: 0.0
os: linux

files:
  - source: /
    destination: /var/www/html
```

![AppSpec File Configuration](images/image0010)

### Important Note

Deployment will fail if the `appspec.yml` file:

- Is missing.
- Is placed inside another folder.
- Is not located in the repository root directory.

---

## 11. Configuring the Deployment Stage in CodePipeline

1. Open AWS CodePipeline.
2. Edit the existing pipeline.
3. Add a **Deploy** stage.
4. Select **AWS CodeDeploy** as the deployment provider.
5. Select the previously created:
   - CodeDeploy Application
   - Deployment Group
6. Save the pipeline configuration.

![Pipeline Deployment Stage Configuration](images/image0011)

---

## 12. Executing the Deployment Pipeline

1. Commit and push changes to the GitHub repository.
2. Monitor the pipeline execution progress.
3. Observe artifact transfer from:
   - GitHub
   - CodeBuild
   - CodeDeploy
   - EC2 Instance

![Pipeline Execution Progress](images/image0012)

---

## 13. Monitoring Deployment Status

1. Open the AWS CodeDeploy Console.
2. Navigate to **Deployments**.
3. Monitor deployment lifecycle events:
   - Download Bundle
   - BeforeInstall
   - Install
   - AfterInstall
   - ApplicationStart
   - ValidateService
4. Confirm that all deployment phases complete successfully.

![Deployment Status Monitoring](images/image0013)

---

## 14. Verifying Successful Deployment

1. Retrieve the EC2 Public IP Address.
2. Open a web browser.
3. Navigate to:

```text
http://<EC2-Public-IP>
```

4. Verify that the deployed application loads successfully.

![Deployment Verification](images/image0014)

### Important Note

If the application uses localhost references, replace them with the EC2 Public IP address before testing.

---

## 15. Common Deployment Failures

### Deployment Group Cannot Find Target EC2

Possible Cause:

- Deployment Group tags and EC2 tags do not match.

### CodeDeploy Agent Not Running

Possible Cause:

- Agent installation failed.
- Agent service was not restarted.

### AppSpec Validation Error

Possible Cause:

- Missing `appspec.yml`
- Incorrect file location

### Deployment Failure During Revision

Possible Cause:

- Application files were not packaged correctly.
- Deployment instructions are invalid.
